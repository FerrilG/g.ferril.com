Design Portfolio

Research

I conduct routine discovery by browsing successful designers and reading through their portfolios. While doing this, I’m paying attention to their style of writing about design decisions. I also read blogs and other articles on sites such as UX Collective, UX Mastery, UX Blog, Medium Design articles. Combined, these articles help keep my focus and offer different approaches to articulating my design decisions.

Direct user research is difficult because feedback is recieved in various forms and is hard to pin-point usability issues. My primary method of obtaining user insights is through the use of Google Analytics.

Google Analytics

Google Analytics helps me observe where the majority of traffic is coming from and the various facets of my site users found interesting.

The three primary data sources I use to draw conclusions are session duration, traffic source. The drawback to using this source is that I have no way of knowing the why’s behind the data. Altering the site based on these figures could be detrimental, so I’d need to include another source of research.

Colleague Review

I tried to utilize colleague review, however have had little success. This approach presents multiple problems by itself, whether it’s the reviewers schedule, the extent of visibility on collaborative work, or just the sheer interest of looking at a portfolio.

When doing a colleague review, I primarily assess if they understood the listed projects and if there are areas I could elaborate on to complete the whole story.

Job Seeking

Interviewing is the final and most critical way of gaining user insight. This is because I’m working directly with my site’s intended users. Throughout the interview process, users observe and can directly ask questions. Not to mention, walking through projects and verbally communicating the scope of projects myself.

All of these methods combined have allowed me to identify the critical elements of a successful portfolio as well as areas for improvement.

Design Concepts

I try to minimize the concepts of my portfolio. This is because I’ve found it to be unnecessary and/or a time sink due to boiling the ocean and then I wind up rebuilding the framework while I’m down in the weeds.

I have however, had success concepting minor tweaks like the placement of elements and the page layout with very rough wireframes. These are useful when I do commit to cleaning up my code and launching a revised framework.

Versions over time

The visual design of my portfolio is intended to reflect my self-interest of working on applications that deal directly with workforce users. I modeled my portfolio around an admin dashboard. Key components include sidebar navigation, top of page control bar, and component cards that one might observe when using similar applications. I’ve never been attached to specific designs, however I felt that by using this visual representation it would establish consistency and insight.

Enhancements
Section for professional experience

(Version 1 - Fig 2) No where on my site did I display my professional experience. I relied on my resume to communicate this; the issue was that both items should be able to stand on its own and not need to be together.

(Version 2 - Fig 3) I added a section at the bottom of the homepage that mimicked my resume as a timeline. This would allow HR to scan one page without clicking to identify my history

Adding personality

(Version 1 - Fig 2) I felt images assisted users with the ability to build connections to myself without having a relationship.

(Version 2 - Fig 3) I personally felt the display of items was a little excessive so I removed the bulk of them. In doing so the new design adopted a serious tone and didn’t portray my enthusiastic personality

(Version 3 - Fig 4) I have since added additional images back to the site. In doing so I wanted to be more reserved and not so blunt about them. My personal opinion is that it complements the site nicely.

Scannability

(Version 1 - Fig 2) To assist HR and Managers answer the questions they had I tried to provide a high-level overview of myself on the home page. I chose to use drawers within the case studies. This was influenced by users wanting to find information quickly. The downside to this is that it was tedious for users across multiple studies.

(Version 2 - Fig 3) To improve this, I introduced a project overview titled TL;DR (Too Long; Didn’t Read). I felt this would assist HR to browse projects for keywords or areas of design the team needs. Through continuing research, I learned that not everyone knows what TL;DR meant and thus was confusing.

(Version 3 - Fig 4) With this version I have removed the label and just display the overview under the title.

Development

Initially, I used Squarespace to host my work; however, I quickly became frustrated with the customization hurdles. So, I decided to develop and host my portfolio directly. At the time I sought the use of CSM functionality so I could modify the content seamlessly. Due to site complexity implementing such features seemed unnecessary so I just created a single JSON file to use. This file held the majority of the sites content including project content, images, and various variables for rendering. Through time I learned that this approach was not scalable, and I soon deviated away.

I classified each version of my site based on the framework & render style for importing the content.

Version 1 (2017)

This version consisted of a linear file structure that used JavaScript to imitate depth. This site was thrown together quickly as I wasn’t too familiar with PHP. This site functioned but the scalability issues became apparent as I started adding content.
Version 2 (2018)

I chose to utilize a modern site file structure. I have a folder for global site elements (header, footer, page template, etc.) and folders for different content. With this method I was able to focus primarily on the page content and not so much on making it ‘work’. I implemented modular elements to quickly render page content and rely on the cache.
Version 3 (2019)

Version 3 utilizes a similar framework to 2.0, except with additional depth. I implemented configuration settings for the ability to turn projects on or off as well as a locking confidential material. Through this implementation I have expanded my working knowledge with back-end scripts and the benefits of using them.
Version 4 (2020)

Version 4 is built upon Angular 10 using the latest framework for modular application construction. My previous versions imitated this structure and now it's implemented.
